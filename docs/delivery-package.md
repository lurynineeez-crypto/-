# 智能种植方舱可视化管理系统交付说明

## 1. 项目目标

本系统面向 38m2 种植方舱，基于现有传感器和设备状态点位，建设一套可视化、可追溯、可演示、可扩展的种植管理原型。

系统重点不是新增硬件，而是把已有硬件数据转化为种植管理能力：

- 实时监控方舱环境、设备、安全和报警状态。
- 用三维方舱展示点位分布，并预留 GLTF 模型替换。
- 围绕批次、阶段、菌包、层架、采收和清舱形成种植闭环。
- 提供环境评分、均匀性、风险、预测、放行、复盘和推荐能力。
- 支持后续接入 MySQL、InfluxDB、Redis、MQTT 和真实设备协议。

## 2. 硬件边界

本原型严格使用文档中已有硬件点位，不新增传感器或设备。

已覆盖的硬件/点位类型包括：

- 温湿度、CO2、光照。
- 料温、基质湿度。
- 不新增风速、压力、氧气等独立环境辅助点位；通风、压差和空气状态相关判断由 CO2 回落、温湿度波动、新风设备状态和人工巡检记录推导。
- 空调、新风、加湿、LED 等设备状态点。
- 烟雾、水浸、门磁、PIR、VOC/异味等安全点。
- 电流、漏电、配电箱温度等电气安全点。
- 数据采集模块、边缘网关和点位台账。

当前模拟台账按 68 个点位组织，后续需要现场复核安装坐标与设备协议字段。

## 3. 技术架构

前端：

- Vue 3 + TypeScript。
- Three.js 三维方舱。
- GLTFLoader 预留真实模型加载。
- ECharts 趋势图表。
- WebSocket 实时数据展示。

后端：

- Node.js NestJS。
- REST API 模拟点位、报警、总览和三维坐标。
- Socket.IO/WebSocket 模拟实时推送。

后续接入：

- MySQL：批次、用户、权限、工单、审批、配置、采收、巡检。
- InfluxDB：温湿度、CO2、光照、设备状态等时序数据。
- Redis：缓存、会话、实时状态、消息中转。
- MQTT：设备消息、边缘网关数据、设备状态订阅。
- 设备协议：待硬件同事确认。

## 4. 导航收口

当前导航已整理为 6 个业务入口：

- 首页/总控：驾驶舱、总控大脑、用户中心、三维方舱、大屏。
- 种植作业：批次、阶段、菌包、日常任务、移动巡检、采收、异常、日历、工艺、清舱。
- 质量放行：质量管控、主任工作台、策略沙盘、专业分析、风险决策、经验优化、复盘会议、标准协同。
- 运维设备：环境、报警、维护校准、配置、点位档案、SOP、联动控制、历史查询。
- 报表交付：经营分析、报表、审计、追溯、工程交付、联调、交付验收、汇报演示、演练、交付总览。
- 系统管理：权限、设置、多端通知、演示控制台、全局检索、解释自检、报表权限。

## 6. API 草案

当前原型已经预留以下 API：

- `GET /api/sensors`：传感器点位台账和当前值。
- `GET /api/dashboard/summary`：首页总览数据。
- `GET /api/alarms`：报警事件。
- `GET /api/3d/points`：三维点位坐标。
- `WS /telemetry`：实时遥测推送。

后续建议扩展：

- `/api/batches`：批次档案、阶段、采收、复盘。
- `/api/work-orders`：工单、SOP、处理闭环。
- `/api/quality`：CCP、CQA、CAPA、阶段放行。
- `/api/reports`：日报、周报、批次报告、验收报告。
- `/api/approvals`：审批流、批次锁定、推荐单生效。
- `/api/users`：用户、角色、权限、操作审计。

## 7. 数据库设计草案

MySQL 建议表：

- `users`：用户与角色。
- `sensor_points`：点位台账、类型、位置、三维坐标。
- `batches`：种植批次。
- `growth_stages`：生长阶段模板。
- `work_orders`：工单。
- `alarms`：报警事件。
- `inspections`：巡检记录。
- `approvals`：审批流。
- `quality_records`：质量记录、CCP/CQA/CAPA。
- `harvest_records`：采收和分级。
- `reports`：报表归档。
- `audit_logs`：操作审计。

InfluxDB measurement 建议：

- `environment_temperature_humidity`。
- `co2_readings`。
- `light_readings`。
- `substrate_readings`。
- `device_status`。
- `safety_events`。
- `electrical_status`。

Redis key 建议：

- `realtime:sensor:{id}`。
- `dashboard:summary`。
- `ws:session:{userId}`。
- `alarm:unhandled`。
- `task:queue:{role}`。

MQTT topic 建议：

- `cabin/{cabinId}/sensor/{pointId}/telemetry`。
- `cabin/{cabinId}/device/{deviceId}/status`。
- `cabin/{cabinId}/alarm/event`。
- `cabin/{cabinId}/gateway/status`。

## 8. GLTF 模型接入约定

前端已预留 GLTF 加载：

- 推荐文件路径：`frontend/public/models/cabin.gltf`。
- 模型坐标系需要与点位 `x/y/z` 坐标统一。
- 点位绑定建议使用 `sensor_points` 表中的 `position` 字段。
- 模型加载失败时保留当前 Three.js 方舱兜底模型。
- 现场交付前需要复核层架、设备、门、风口和安全点位坐标。
